**CAN 总线**（Controller Area Network Bus：控制器局域网总线）是一种**多主机的串行通信总线标准**，由 **Bosch 公司在1983年**为汽车系统设计，后来广泛应用于工业控制、医疗设备、电梯、电动车、机器人等多个嵌入式场合。

## CAN 总线的核心特点

| 特点               | 说明                                               |
| ------------------ | -------------------------------------------------- |
| **多主通信**       | 所有节点都可以主动发起通信（不像UART那样主从结构） |
| **优先级仲裁机制** | 通过报文ID自动仲裁，无冲突丢包                     |
| **差分信号**       | CANH 和 CANL 两根线，抗干扰强，适合工业环境        |
| **无需主机**       | 任何节点可上电即用，方便扩展、灵活部署             |
| **实时性强**       | 报文短小，仲裁机制确保高优先级快速传输             |
| **错误检测能力强** | 带 CRC 校验、位填充、应答机制等                    |

**多主通信**
所有节点都可以主动发起通信（不像 UART 那样主从结构，**UART 是 一对一的，不支持多节点同时连接通信**。）

**优先级仲裁机制**
当两个或多个节点**同时发送**数据时，CAN 总线使用报文的 **ID 来自动仲裁优先级**：
报文 ID 越小，优先级越高；
仲裁过程基于**位级竞争机制**（0胜于1）；
整个过程不产生冲突、不会丢帧！
高优先级数据（如刹车状态）会更快传输；
低优先级节点会自动延后，不会干扰关键任务。

**差分信号**
CANH 和 CANL 两根线，抗干扰强，适合工业环境
两根线电压**反向波动**（一个升高，一个降低）
接收端取两线电压**差值**来判断信号抗干扰能力强：即使两根线同时受到电磁干扰，差值不变；
适合恶劣环境：工厂、电动车、医疗设备等电磁复杂场合；
长距离传输也稳：数十米到上百米都可以（配终端电阻）

**无需主机**
不像 SPI / I²C / Modbus 这类通信需要一个“主设备”统一调度，CAN 的每个节点都是**平等的参与者**，没有“主控”依赖。加一个节点，不用改整个通信结构；
某个模块故障，也不影响其他节点运行；
系统更**容错**、更**灵活维护**。

**实时性强**
报文短小，仲裁机制确保高优先级快速传输
报文最小只要 **几个字节**（最大 8 字节，CAN FD 扩展到 64）；
短报文 + 高速传输（一般 500Kbps~1Mbps）= 快速响应；
仲裁机制保证重要数据优先发出，**关键数据不会“被堵住”**。

> 虽然差分信号各设备不共地可以使用
> 但是这可能会让收发器承受较高的共模电压
> 在实际应用中，为了安全和稳定，CAN通信一般还是会共地的
> 此外，如果除了传递信号，还需要为设备传递供电电源那么就需要再加一根电源正的线路电源正，电源负(GND)，CANH，CANL这 4根线一起构成完整的供电和通信线路

**高速 CAN(ISO11898)：125K ~ 1Mbps，< 40m**
**低速 CAN(ISO11519)：10K ~ 125Kbps，< 1km**

**异步，无需时钟线，通信速率由设备各自决定**

**半双工，可挂载多设备，多设备同时发送数据时通过仲裁判断先后顺序**

**可实现广播式和请求式两种传输方式**

## 架构组成

**物理层（PHY）**：由 CAN 收发器芯片（如 TJA1050）完成差分驱动；
**数据链路层**：由 MCU 内部的 CAN 控制器（如 STM32 的 bxCAN）完成帧格式、仲裁、错误检测等；
**CAN 节点** = MCU（带 CAN 控制器）+ CAN 收发器 + 软件协议栈

## CAN 硬件电路

每个设备通过 CAN 收发器挂载在 CAN 总线网络上
CAN 控制器引出的 TX 和 RX 与 CAN 收发器相连，CAN 收发器引出的 CAN_H 和 CAN_L 分别与总线的 CAN H 和 CANL 相连
高速 CAN 使用闭环网络，CA_H 和 CA_L 两端添加 12Ω 的终端电阻
低速 CAN 使用开环网络，CAN_H和 CA_L 其中一端添加 2.2kΩ 的终端电阻
![canyjdl](/public/image/嵌入式/通信协议/canyjdl.png)

> CAN 的 TX 连 TX，RX 连 RX，二者无需交叉
> 在闭环 CAN 中：终端电阻作用有二，一是防止回波反射，二是在没有设别操作时将两根差分线的电压"收紧"，使其电压一致
> 回波反射：如果不加终端电阻，会导致信号波形在线路终端反射，进而干扰原始信号
> 这里与 I2C 类似，输出 0 是拉低信号，输出 1 时，他无需对总线进行任何操作，由终端电阻归位为 1
> 在开环 CAN 中：由于电路没有闭环，所以终端电阻没有"收紧"作用。

## CAN 电平标准

CAN 总线采用差分信号，即两线电压差(V<sub>CAN_H</sub>>-V<sub>CAN_L</sub>>)传输数据位
高速 CAN 规定:
电压差为 0V 时表示逻辑 1(隐性电平)
电压差为 2V 时表示逻辑 0(显性电平)
低速 CAN 规定:
电压差为 -1.5V 时表示逻辑 1(隐性电平)
电压差为 3V 时表示逻辑 0(显性电平)

## CAN 收发器-TJA1050(高速 CAN)

![cansfq](/public/image/嵌入式/通信协议/cansfq.png)

> **主要组件**：
>
> - TJA1050T：CAN 收发器芯片，负责将 CAN 控制器的数据信号转换为 CAN 总线的差分信号（CAN_H 和 CAN_L），并提供温度保护功能。
> - 驱动器 (DRIVER)：将 TXD（传输数据输入）信号转换为 CAN 总线上的差分信号。当 TXD 输出 1 时 DRIVER 会让 上下两个场效应管都断开，也就是不对总线进行任何操作，总线在外部终端电阻的收紧作用下拉高为 1；当 CAN 总线未被驱动时，0.5Vcc 电压通过 25 kΩ 电阻对 CAN_H 和 CAN_L 施加偏置。让二者默认对地电平都在 2.5V(5V 的 0.5 倍) 左右。当 TXD 输出 0 时 DRIVER 会让 上下两个场效应管都导通，二者分别将 CAN_H 和 CAN_L 拉高拉低形成压差，让 CAN 总线呈现显性 0 的操作。当 TXD 输出 1 的时候，VCC 上拉防止输入引脚电平不确定造成误操作的情况。
> - 接收器 (RECEIVER)：从 CAN 总线读取差分信号(有压差是 1，无压差是 0)并反向（0 输出 1，1 输出 0）输出到 RXD（接收数据输出）。
> - Dominant Time-out Timer：防止总线长时间处于主导状态，保护电路。（比如 TXD 异常，始终输出显性电平 0 ，始终拽着总线不放，这回导致整条总线瘫痪，并且没有设备可以阻止这一情况，所以收发器在这里做了一层保险，如果出现上述问题，收发器会自动取施放总线，防止总线瘫痪。）
> - 参考电压 (REFERENCE VOLTAGE)：通过 Vref 引脚提供稳定的参考电压。
>
> **电源与接地**：
>
> - VCC（电源电压）通过 200 µA 电流限制电阻连接到电路。
> - GND（接地）在多个点连接，确保电路的稳定性和信号参考。
>
> **保护机制**：
>
> - 温度保护电路在芯片过热时工作，防止损坏。
> - 二极管和电阻（25 kΩ）用于 CANH 和 CANL 线的保护。
>
> **信号流**：
>
> - TXD（引脚 1）输入信号经过驱动器处理后，输出到 CANH（引脚 7）和 CANL（引脚 6）。
> - RXD（引脚 4）从接收器获取总线数据，传输回 CAN 控制器。
> - S（引脚 8）为模式选择输入，用于切换高速度模式或静音模式。
>

## CAN 总线的帧格式

![canzxzgs](/public/image/嵌入式/通信协议/canzxzgs.png)

### 数据帧

![cansjz](/public/image/嵌入式/通信协议/cansjz.png)

**标准帧**

| 字段名              | 位数     | 作用说明                                                     |
| ------------------- | -------- | ------------------------------------------------------------ |
| **SOF**             | 1        | Start of Frame，帧起始标志，一个显性位（Dominant）           |
| **Identifier (ID)** | 11       | 报文标识符(标识是什么数据)，决定优先级，越小优先级越高       |
| **RTR**             | 1        | Remote Transmission Request，远程帧请求位（0表示数据帧，1表示远程帧） |
| **IDE**             | 1        | Identifier Extension，标志是否为扩展帧（0表示标准帧）        |
| **r0**              | 1        | 保留位                                                       |
| **DLC**             | 4        | Data Length Code，指示数据段有几字节（0~8）                  |
| **Data**            | 0~64字节 | 实际的数据内容（最多8字节，CAN FD可到64）                    |
| **CRC**             | 15       | CRC校验码                                                    |
| **CRC定界符**       | 1        | CRC结束标志位，为应答位前后发送方和接收方释放总线留下时间，发送方释放总线 |
| **ACK槽**           | 1        | ACK Slot，接收方会拉低此位响应发送方                         |
| **ACK定界符**       | 1        | ACK段结束标志，为应答位前后发送方和接收方释放总线留下时间，接收方释放总线 |
| **EOF**             | 7        | End of Frame，帧结束标志，7个隐性位（Recessive）             |

**扩展帧**

| 字段名              | 位数 | 说明                                                        |
| ------------------- | ---- | ----------------------------------------------------------- |
| **Identifier (ID)** | 11   | 基础 ID（跟标准帧一样）                                     |
| **SRR**             | 1    | Substitute Remote Request，扩展帧用来代替 RTR，固定为显性位 |
| **IDE**             | 1    | 此时 IDE = 1，表示是扩展帧                                  |
| **Extended ID**     | 18   | 扩展的 18 位标识符                                          |
| **RTR**             | 1    | 同标准帧，表示是否为远程帧                                  |
| **r0**              | 1    | 保留位                                                      |

> 💡除了前半段结构不同，扩展帧的**控制段、数据段、CRC段、ACK段和EOF**与标准帧结构完全一致。
>
> CAN 协议的收发双方是同步的，也就是或发送方发送一个高低电平，接收方就接收一个高低电平，而不是等发送方发送完成后接收方才开始接收；发送方与接收方共同构成了一个完整波形。

### 遥控帧

![canykz](/public/image/嵌入式/通信协议/canykz.png)

| 区域      | 标准帧（上） | 扩展帧（下） | 说明                                            |
| --------- | ------------ | ------------ | ----------------------------------------------- |
| **SOF**   | 1 bit        | 1 bit        | Start of Frame，帧起始标志                      |
| **ID**    | 11 bit       | 11+18 bit    | Identifier 标识符，决定报文优先级               |
| **SRR**   | ✖            | 1 bit        | Substitute Remote Request，扩展帧固定为隐性位   |
| **IDE**   | 1 bit        | 1 bit        | 0=标准帧，1=扩展帧                              |
| **RTR**   | 1 bit        | 1 bit        | Remote Transmission Request = **1**，表示远程帧 |
| **r1/r0** | 1 bit        | 2 bits       | 保留位，未来扩展用，固定为显性                  |
| **DLC**   | 4 bit        | 4 bit        | 要求返回的数据长度（最多8字节）                 |
| **Data**  | ❌ 无         | ❌ 无         | 远程帧没有数据段                                |
| **CRC段** | 15+1 bit     | 15+1 bit     | 校验码+分隔符                                   |
| **ACK段** | 2 bit        | 2 bit        | 确认槽位，接收方回应                            |
| **EOF**   | 7 bit        | 7 bit        | 报文结束标志，7个隐性位                         |

> 当请求和返回同时发生时，数据帧拥有更高的优先级

### 错误帧

总线上所有设备都会监督总线的数据，一旦发现“位错误”或“填充错误”或“CRC错误”或“格式错误”或“应答错误” ，这些设备便会发出错误帧来破坏数据，同时终止当前的发送设备
![canczw](/public/image/嵌入式/通信协议/canczw.png)

> 主动：强制打断当前帧传输，通知所有节点**这里有错误，帧无效**（默认）
>
> 被动：告诉其他节点“我检测到了错误”，**但不会主动打断总线**
>
> 主动错误触发的太频繁，说明该设备不可靠，设备就会进入被动错误状态，处于被动错误状态的设备检测出错误时，会连续发 6 个隐性位(就是不碰总线)，但是会破坏自己发送的数据，不影响其他设备。
>
> 图中 0-6 表示的是错误标志延长的时间，因为当一个设备发出错误标志时，可能会带动其他设备也发出错误标志，此时错误标志一共可能就不止 6 位了；所以这里的 0-6 标识延迟的时间。
>
> 错误帧会叠加在数据帧上（类似"线与"，主动错误将总线拉低到 0，这样所有的 1 遇到 0 会变成 0）

### 过载帧

当接收方收到大量数据而无法处理时，其可以发出过载帧，延缓发送方的数据发送，以平衡总线负载，避免数据丢失
![cangzz](/public/image/嵌入式/通信协议/cangzz.png)

### 帧间隔

将数据帧和遥控帧与前面的帧分离开，给节点提供时间来处理接收到的数据、恢复同步或准备发送下一帧。
![canzjg](/public/image/嵌入式/通信协议/canzjg.png)

> 连续发送数据帧时，其中间会有一小段帧间隔
> 协议中，帧间隔（Interframe Space）期间的总线控制并不是由某个单一的“总监”或特定节点主动管理，而是由协议本身和所有节点共同遵循的分布式机制来协调。

## 位填充

**位填充规则：**发送方每发送 5 个相同电平后，自动追加一个相反电平的填充位，接收方检测到填充位时，会自动移除填充位，回复原始数据。

例如：
即将发送：     100000110     10000011110       0111111111110
实际发送：     1000001110   1000001111100  011111011111010
实际接收：     1000001110   1000001111100  011111011111010
移除填充后： 100000110     10000011110       0111111111110

位填充作用：
增加波形的定时信息，利于接收方执行“再同步”，防止波形长时间无变化，导致接收方不能精确掌握数据采样时机
将正常数据流与“错误帧”和“过载帧”区分开，标志“错误帧”和“过载帧”的特异性
保持CAN总线在发送正常数据流时的活跃状态，防止被误认为总线空闲

> 他是接收方的第一步处理，也是发送方的最后一步处理；
>
> 例子：发送方需要发送一个数据帧，他的数据段有 8 个字节，也就是 64 位，且数据全部都是 1；如果直接发送这个波形，总线在数据段时会呈现出 64 位的 1，这个时间是比较长的，如果接收方和发送方始终稍有偏差，就会导致接收到的数据不准确；位与位之间没有变化，就导致位与位之间没有分界点了；接收方没法利用数据的跳变沿确定采样点是不是在同一位的中间，也无法得知采样点的位置是否准确，更没法依据跳变沿，执行再同步，进行误差补偿了；所以长时间不变的波形，容易导致接收出错；为了避免这个问题我们就让发送方发送数据时候强行变化一下，加入位填充后发送方不可能出现连续 5 个以上的相同位，数据多跳变跳变，数据位的边界才会更加清晰，才利于接收方确定采样点的准确位置。
>
> CAN 总线规定，如果连续出现 11 个隐性的 1，即认为总线空闲。

## 接收方数据采样

CAN总线没有时钟线，总线上的所有设备通过约定波特率的方式确定每一个数据位的时长
发送方以约定的位时长每隔固定时间输出一个数据位
接收方以约定的位时长每隔固定时间采样总线的电平，输入一个数据位理想状态下，接收方能依次采样到发送方发出的每个数据位，且采样点位于数据位中心附近

**接收方数据采样异常**
接收方以约定的位时长进行采样，但是采样点没有对齐数据位中心附近。
接收方刚开始采样正确，但是时钟有误差，随着误差积累，采样点逐渐偏离。

## 位时序

为了灵活调整每个采样点的位置，使采样点对齐数据位中心附近，CAN 总线对每一个数据位的时长进行了更细的划分，分为同步段(SS)、传播时间段(PTS)、相位缓冲段1(PBS1)和相位缓冲段2(PBS2)，每个段又由若干个最小时间单位(Tq)构成
![canwsx](/public/image/嵌入式/通信协议/canwsx.png)

> 各个段的时间可以由我们自行指定，范围如图中右边所示。
> 如果数据跳变沿发生在 SS 段，则表面当前设备与波形达成同步；如果数据跳变沿没有发生在 SS 段，则需要调整当前设备的位时序。

## 硬同步

每个设备都有一个位时序计时周期，当某个设备(发送方)率先发送报文其他所有设备(接收方)收到SOF的下降沿时，接收方会将自己的位时序计时周期拨到SS段的位置，与发送方的位时序计时周期保持同步
硬同步只在帧的第一个下降沿(SOF下降沿)有效
经过硬同步后，若发送方和接收方的时钟没有误差，则后续所有数据位的采样点必然都会对齐数据位中心附近
![canytb](/public/image/嵌入式/通信协议/canytb.png)

> SOF 下降沿：逻辑电平的下降沿，差分电平的张开沿。

## 再同步

若发送方或接收方的时钟有误差，随着误差积累，数据位边沿逐渐偏离 SS 段则此时接收方根据再同步补偿宽度值(SJW)通过加长 PBS1段，或缩短 PBS2 段，以调整同步
再同步可以发生在第一个下降沿之后的每个数据位跳变边沿
![canztb](/public/image/嵌入式/通信协议/canztb.png)

> SJW：指的是再同步补偿宽度的最大值，当实际误差值小于 SJW 时，误差几个 Tq 就补偿几个 Tq；当 SJW 小于实际误差值时，不管误差多大，实际都只会补偿 SJW 个 Tq
>
> SJW 的值要根据实际情况进行选择，时钟误差小可以设计为 1-2Tq，时钟误差大可设置为 3-4Tq。

## 波特率计算

波特率= 1 / 一个数据位的时长 = 1 / (TsS + TPTS + TPBS1 + TPBS2)
例如: SS = 1Tq，PTS = 3Tq，PBS1 = 3Tq，PBS2 = 3Tq, Tq = 0.5us
波特率= 1 / (0.5us + 1.5us + 1.5us + 1.5us) = 200kbps

## 仲裁

CAN总线只有一对差分信号线，同一时间只能有一个设备操作总线发送数据若多个设备同时有发送需求，该如何分配总线资源?
解决问题的思路:制定资源分配规则，依次满足多个设备的发送需求，确保同一时间只有一个设备操作总线

### 先占先得

若当前已经有设备正在操作总线发送数据帧 / 遥控帧，则其他任何设备不能再同时发送数据帧 / 遥控帧(可以发送错误帧 / 过载帧破坏当前数据)。
任何设备检测到连续 11 个隐性电平，即认为总线空闲，只有在总线空闲时设备才能发送数据帧 / 遥控帧。
一旦有设备正在发送数据帧 / 遥控帧，总线就会变为活跃状态，必然不会出现连续 11 个隐性电平，其他设备自然也不会破坏当前发送。
若总线活跃状态其他设备有发送需求，则需要等待总线变为空闲，才能执行发送需求。

### 非破坏性仲裁

若多个设备的发送需求同时到来或因等待而同时到来，则 CAN 总线协议会根据 ID 号(仲裁段)进行非破坏性仲裁，ID 号小的(优先级高)取到总线控制权，ID 号大的(优先级低)仲裁失利后将转入接收状态，等待下一次总线空闲时再尝试发送
实现非破坏性仲裁需要两个要求
线与特性：总线上任何一个设备发送显性电平 0 时，总线就会呈现显性电平 0 状态，只有当所有设备都发送隐性电平 1 时，总线才呈现隐性电平 1 状态，即：0 & X & X = 0，1 & 1 & 1 = 1
回读机制：每个设备发出一个数据位后，都会读回总线当前的电平状态以确认自己发出的电平是否被真实地发送出去了，根据线与特性，发出 0 读回必然是 0，发出 1 读回不一定是 1

**非破坏性仲裁过程**
数据位从前到后依次比较，出现差异且数据位为1的设备仲裁失利
![canfphxzcgc](/public/image/嵌入式/通信协议/canfphxzcgc.png)

> 位填充不会影响仲裁的结果

**数据帧和遥控帧的优先级**
数据帧和遥控帧ID号一样时，数据帧的优先级高于遥控帧
![cansjzykzyxj](/public/image/嵌入式/通信协议/cansjzykzyxj.png)

**标准格式和扩展格式优先级**
标准格式11位ID号和扩展格式29位ID号的高11位一样时，标准格式的优先级高于扩展格式（SRR必须始终为1，以保证此要求）

## 错误状态

主动错误状态的设备正常参与通信并在检测到错误时发出主动错误帧
被动错误状态的设备正常参与通信但检测到错误时只能发出被动错误帧
总线关闭状态的设备不能参与通信
每个设备内部管理一个TEC和REC，根据TEC和REC的值确定自己的状态
![canczwt](/public/image/嵌入式/通信协议/canczwt.png)

> TEC(Transmit Error Counter)：发送错误计数器，用于跟踪节点在发送数据帧时发生的错误数量的计数器；每当节点检测到自己发送的帧出现错误（例如位错误、CRC 错误或仲裁丢失），TEC 会增加；当成功发送帧时，TEC 减少 1（但不会低于 0）。
>
> REC(Receive Error Counter)：接收错误计数器，用于跟踪节点在接收数据帧时检测到的错误数量的计数器；当节点接收到有错误的帧（例如校验错误或格式错误）时，REC 会增加；当成功接收帧时，REC 减少 1（但不会低于 0）。
